# Implementation Plan: Die Rolling Support (066-die-rolling)

**Branch**: `066-die-rolling` | **Date**: 2026-03-10 | **Spec**: [specs/066-die-rolling/spec.md]
**Input**: Feature specification from `/specs/066-die-rolling/spec.md`

## Summary

Implement a robust die-rolling system for Codex-Cryptica that supports standard polyhedral dice, advanced logic (Keep Highest/Lowest, Exploding), and contextual display (Oracle Chat vs. Modal Log). The core logic will be encapsulated in a standalone `dice-engine` package to adhere to the Library-First principle.

## Technical Context

**Language/Version**: TypeScript 5.x, Node.js 20+  
**Primary Dependencies**: Svelte 5 (Runes), `idb` (for history), Web Crypto API (for fair randomization)  
**Storage**: IndexedDB (via `idb`) for session-based roll history  
**Testing**: Vitest (Unit/Integration), Playwright (E2E)  
**Target Platform**: Web (SvelteKit)  
**Project Type**: Monorepo (Package + Web UI)  
**Performance Goals**: <300ms result display, 100% parsing accuracy  
**Constraints**: Client-side only (Privacy), individual results transparency  
**Scale/Scope**: Support for standard RPG dice (d4-d100) and advanced modifiers.

## Constitution Check

_GATE: Must pass before Phase 0 research. Re-check after Phase 1 design._

1. **Library-First**: CORE logic MUST be in `packages/dice-engine`. `apps/web` MUST only handle UI.
2. **Test-Driven Development**: `dice-engine` MUST have 100% coverage for formula parsing and result calculation.
3. **Simplicity & YAGNI**: Research existing dice parsing libraries (e.g., `dice-roller-parser`) to avoid custom parser maintenance if possible.
4. **AI-First Extraction**: N/A for this feature (purely arithmetic/logic).
5. **Privacy & Client-Side**: All randomization and calculation MUST happen in the browser.
6. **User Documentation**: MUST add entry to `apps/web/src/lib/config/help-content.ts` and potentially a `FeatureHint`.
